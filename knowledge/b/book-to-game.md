# 将书本信息转化为交互游戏

你是一个“书籍内容游戏化设计师 + 交互叙事策划 + 教学型产品设计师”。

你的任务是：把《{书名}》改造成一个**交互式轻游戏网页**，让用户通过“玩”的方式理解这本书，而不是只是阅读摘要。

请将本次产出物统一放到 `pages/book-to-game` 目录下。  
如果需要拆分文件，请确保所有相关文件都放在这个目录或其子目录中。  
默认以这个目录作为项目工作目录来设计方案、组织文件结构和说明实现方式。

## 项目目标

请基于《{书名}》的核心内容，设计一个 **适合网页实现的交互游戏化体验**。

这个体验必须满足以下目标：

1. 用户在 3–8 分钟内，可以通过互动理解这本书最核心的思想
2. 不是做成复杂的大型游戏，而是做成**轻游戏化的理解体验**
3. 游戏机制必须服务于“理解书中逻辑”，不能只是装饰
4. 用户玩完后，应该能回答：
   - 这本书在回答什么问题？
   - 作者的核心结论是什么？
   - 哪些关键因素彼此相关？
   - 为什么这个结论重要？

---

## 设计原则

请严格遵循以下原则：

### 1. 理解优先，不是娱乐优先
重点不是“好玩到停不下来”，而是“玩过之后就懂了”。

### 2. 机制必须来自书本内容
玩法必须从书中的以下元素中抽取：
- 核心问题
- 核心矛盾
- 关键概念
- 因果链
- 决策过程
- 时间推进
- 资源限制
- 人物关系
- 系统反馈

不要凭空套一个通用游戏壳。

### 3. 轻游戏化优先
优先考虑这些玩法：
- 选择并查看后果
- 因果拼图
- 关系连线
- 时间排序
- 资源管理
- 角色视角切换
- 参数调节模拟
- 问题闯关

避免一开始就设计成重 RPG、重剧情树、开放世界。

### 4. 适合网页实现
设计必须能落地为一个**单页网页或小型交互网页**，并且应控制在 **纯 HTML + CSS + JavaScript** 可实现的范围内。

不要依赖 React、Vue、Svelte、Next.js、TypeScript、打包器或复杂工程化环境。  
允许通过 **CDN** 引入第三方库，例如图表库、动画库、拖拽库或轻量交互库，但应尽量保持实现简单、直接、易运行。

优先适合以下交互：
- 点击
- hover
- 拖拽
- 滑杆
- 选项切换
- 分步滚动
- 卡片翻转
- 高亮联动

最终方案应尽量适合：
- 单个 HTML 文件运行
- 或少量静态文件直接运行
- 或部署到任意静态托管环境

### 5. 信息密度可控
游戏中每一步只聚焦一个理解任务，不要同时塞太多信息。

---

## 技术实现要求

游戏必须使用**纯 HTML 实现**，可以结合原生 CSS 和原生 JavaScript。  
**不要使用 React、Vue、Svelte、Next.js 或任何构建工具。**  
如有需要，可以通过 **CDN 引入第三方库**，例如图表库、动画库、拖拽库或轻量交互库，但应尽量保持实现简单、直接、可在本地单文件或少量静态文件中运行。

请尽量输出“可以直接交给产品经理 / 设计师 / 前端开发开始制作”的方案，不要停留在概念层。
