# Chore Issue 模板

## 高密度摘要

- **一句话结论**：Chore Issue 应明确维护工作的触发原因、范围、产物、执行步骤和回滚方式，而不只写一个操作名称。
- **核心机制**：用可检查的产物和完成标准约束维护任务，让升级、迁移、清理和配置工作具备可恢复、可验证的执行边界。
- **判断入口**：依赖升级、迁移、清理、配置、文档、合规或例行维护时使用；直接交付用户能力使用 Feat，纠正错误行为使用 Fix。
- **常见误区**：不要把所有技术工作都归为 Chore，也不要忽略维护操作对数据、兼容性和正在运行流程的影响。
- **相关文档**：[Feat Issue 模板](./template-project-issue-feat.md)、[Fix Issue 模板](./template-project-issue-fix.md)。

## 使用说明

- 创建时填写触发原因、范围、预期产物、执行步骤和完成标准。
- 迁移、配置或依赖升级需要记录中断风险、备份要求和回滚方式；不可回滚时必须明确说明。
- `## 执行结果` 用于记录实际结果和与计划的差异，不要在执行前假设成功。
- `## 验证` 在创建时保持为空，只在检查产物和受影响流程后填写。

## 模板

```markdown
---
type: chore
status: backlog
urgent: false
endDate:
---

# 一句话描述维护工作

## 背景与触发

- 为什么现在需要做：【填写】
- 触发来源：【依赖升级、迁移、清理、配置、文档、合规或其他】

## 范围与产物

- 包含：【填写】
- 不包含：【填写】
- 预期产物：【代码、配置、文档、迁移结果或其他】

## 执行步骤

- [ ] 【填写一个可独立确认的步骤】

## 完成标准

- [ ] 目标产物已经生成或更新。
- [ ] 受影响流程已经验证。
- [ ] 临时文件、兼容措施或后续工作已经处理或记录。

## 风险与回滚

- 中断或兼容风险：【确认不存在时写“无”；尚未评估写“待确认”】
- 数据或配置备份要求：【确认不需要时写“无”；尚未评估写“待确认”】
- 回滚方式：【不可回滚时明确说明】

## 执行结果

- 当前状态：待执行
- 实际结果：【完成后填写，包括与计划的差异】

## 验证

## pending 原因

## 改动Prompt
```
