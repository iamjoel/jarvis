# Fix Issue 模板

## 高密度摘要

- **一句话结论**：Fix Issue 应保存从问题现象、复现证据到根因、解决方案和回归验证的完整因果链。
- **核心机制**：创建时先记录已知事实，调查后再确认根因；进入 `done` 前用证据连接“为何发生、改了什么、如何证明有效”。
- **判断入口**：实际行为与预期不一致时使用；复杂、反复出现、线上或高严重度问题需要填写完整根因、逃逸原因、监控和回滚信息。
- **常见误区**：不要把猜测写成已确认根因，不要把代码 diff 当作解决方案，也不要用“已修复”代替验证结果。
- **相关文档**：[Feat Issue 模板](./template-project-issue-feat.md)、[Chore Issue 模板](./template-project-issue-chore.md)。

## 使用说明

- 创建时至少填写问题现象、预期行为、影响范围，以及复现步骤或现有证据。
- 根因未知时保留“待确认”，不得为了填满模板而编造结论。
- 正常进入 `done` 前，根因应标记为“已确认”或“较大概率”并附证据；解决方案、回归范围和实际验证结果必须完整。
- 如果只有临时规避方案且根因仍未知，只有在明确不确定性、验证规避效果，并记录一个具体的后续调查 Issue ID 后才能关闭。
