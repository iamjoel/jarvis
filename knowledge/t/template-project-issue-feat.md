# Feat Issue 模板

## 高密度摘要

- **一句话结论**：Feat Issue 应先固定用户价值、范围和可验证的完成标准，再在处理过程中补齐方案与关键决策。
- **核心机制**：把“为什么做”“做什么”和“怎样算完成”分开，减少范围漂移，也避免用实现细节代替用户结果。
- **判断入口**：新增用户能力、改变现有行为或交付新的业务价值时使用；纯维护工作使用 Chore，纠正错误行为使用 Fix。
- **常见误区**：不要在创建时编造尚未确定的方案，也不要把任务清单当作完成标准；完成标准必须描述可观察结果。
- **相关文档**：[Fix Issue 模板](./template-project-issue-fix.md)、[Chore Issue 模板](./template-project-issue-chore.md)。

## 使用说明

- 创建时填写背景、目标、范围和完成标准。
- 方案与关键决策可以在进入 `processing` 后补齐，未知时明确写为“待设计”。
- `## 验证` 在创建时保持为空，只在实际检查完成后记录验证项和结果。
- 进入 `done` 前，完成标准必须逐项有结果，范围变化和关键决策必须已经记录。

### 不包含

- 【填写】

## 完成标准

- 方案：【处理过程中填写】
- 关键决策及原因：【处理过程中填写】

## 风险与发布

- 兼容性或数据影响：【确认不存在时写“无”；尚未评估写“待确认”】
- 发布、灰度或回滚要求：【确认不需要时写“无”；尚未评估写“待确认”】

## 验证
