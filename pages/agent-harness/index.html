Agent Harness
概念 闭环 模式 架构

Execution Layer for Reliable Agents

Agent harness 不是模型本身， 而是让 Agent 可执行、可验证、可交接的那一层。

它把工具、沙盒、测试、评分、日志和上下文移交组织成一个稳定回路，让 Agent 不只是“会回答”，而是能在真实任务里持续交付。

看执行闭环 看四种模式

一句话定义

用标准化环境包住 Agent，并把反馈重新喂回去。

01

可执行

能调工具、跑命令、读写文件，而不只生成文本。

02

可验证

每次动作之后都有测试、评估或可观察反馈。

03

可复现

同一任务在同一 harness 下能重复运行与比较。

04

可交接

长任务中，状态与契约能通过文件持续传递。

Concept

把零散能力“套起来”，Agent 才开始像系统。

输入

任务与上下文

Issue、需求、代码库、浏览器界面、运行中的应用，都是 harness 的观测对象。

中间层

执行环境

沙盒、命令执行、工具注册、超时控制、日志收集与权限边界共同组成 harness。

输出

反馈与评分

测试结果、通过率、失败日志、E2E 验收、handoff 文档，决定 Agent 下一步动作。

Closed Loop

没有 harness，Agent 只能猜；有了 harness，Agent 才能迭代。

Agent
理解任务
执行动作
观察结果
修正策略
1

标准化入口

harness 把“怎么开始”固定下来，例如跑测试、启动浏览器、加载仓库、拉起任务。

2

受控执行

Agent 在权限清晰的环境中调用 shell、文件系统、网页工具或评测框架。

3

反馈回流

pass/fail、日志、截图、评分与异常被结构化返回，而不是散落在外部系统里。

4

继续迭代

Agent 基于同一回路继续修改、验证、再提交，从“一次性输出”转向“闭环生产”。

Four Modes

Harness 常见的四种工作方式

Test Harness

验证代码是否真的被修好

典型场景是修复 Issue、执行补丁、运行测试套件，然后用测试结果判定是否 resolved。

  • 代表场景：SWE-bench、`pytest` / `jest` 封装
  • 核心产物：通过率、错误日志、回归信号

Eval Harness

比较模型能力而不是猜印象分

它统一数据集、模型调用方式与评分规则，让不同模型在同一标准下被测量。

  • 代表场景：`openai/evals`、`lm-evaluation-harness`
  • 核心产物：accuracy、match、model-graded 等指标

Agent Harness

给 Agent 一套能执行真实工作的操作层

它把文件系统、shell、浏览器、截图、鼠标键盘和任务状态串起来，形成工具闭环。

  • 代表场景：Codex 沙盒、Claude Computer Use
  • 核心产物：工具返回值、环境状态、操作记录

Long-running Harness

让长周期任务可以跨 session 接力

重点不只是执行，而是把规划、实现、评估和移交文档做成协议，防止上下文断裂。

  • 代表场景：Planner / Generator / Evaluator 分工
  • 核心产物：spec、contract、bug report、handoff

Long-Running Architecture

当任务很长时，harness 还负责“交接班”。

A

Planner

把模糊目标扩展成可实施的 spec，明确边界、验收标准与技术建议。

B

Generator

按 sprint 逐段实现，每段结束都产出代码、自检与下一段 contract。

C

Evaluator

独立验证是否达标，用 E2E、Playwright 或规则脚本像真实用户一样检查结果。

Handoff Artifacts

文件不是附属物，而是 Agent 之间的协议。

长任务里最常见的失败不是模型不会做，而是 session 之间没有结构化移交。好的 harness 会把 spec、sprint contract、bug report、handoff 文档变成固定产物。

Spec 定义目标与范围
Contract 定义本轮可验证交付
Bug Report 把失败具体化，便于修复
Handoff 让下一个 session 可直接续跑

Takeaway

Harness 的价值，不在于“包一层”，而在于把 Agent 从回答者变成执行者。

当环境、工具、验证与移交都被标准化，Agent 才能持续完成复杂任务，而不是在每次上下文切换后重新开始。
